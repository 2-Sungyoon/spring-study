## 📝 주요 개념 정리
### 메시지 큐

**메시지 큐** : 작업을 바로 처리하지 않고 큐에 쌓아두고, **`순차적`**으로 **`비동기 처리`**하는 시스템

**메시지 브로커** : 좀 더 포괄적인 개념인데, 메시지 큐를 활용해 **`메시지의 흐름`** 전체를 **`관리하고 제어`**하는 시스템(작업을 관리, 메시지의 전달 상태 확인과 같은 역할. 그니까 단순히 큐에 메시지를 쌓는게 아니라, 관리 & 제어하는 거임)

장점 : 

- 확장성
- 처리량 증가
- 대기 시간 감소
- 시스템의 안정성 향상

→ 요청이 몰릴때 터지지 않게 도와줌

그렇다면 둘의 차이점은 ?

- 메시지 큐
    - 작업의 순서 보장이 주 목적
- 메시지 브로커
    - 메시지의 전체적인 흐름을 제어 / 통신 조율이 주 목적
    - 메시지 브로커 = 메시지 큐 + 메시지 라우팅, 밸리데이션, 전달 보장 등의 부가적인 관리 기능

대표적인 기술 :

- RabbitMQ
- Kafka
- ActiveMQ
- SQS

특징 : 

- 무손실(durability) : 메시지 큐에 보관된 메시지는 consumer가 꺼낼 때까지 안전하게 보관됨
    - producer와 consumer는 소비자/생산자 서비스가 다운돼도 메시지를 문제없이 발행/수신할 수 있음
- 비동기 처리 가능
- 시스템 분리 가능

**특징**

- 메시지 처리 후 삭제
- 순차 처리

**장점**

- 메시지 라우팅 가능 (exchange)
- 순서 보장
- 신뢰성 높음

**언제 사용?**

- 알림 처리
- 주문 처리
- 작업 분배

→ 하나씩 처리해야하는 작업에 유리

### Kafka

대용량 데이터 스트리밍 플랫폼

**구조**

- topic / partition 기반

**특징**

- 메시지를 삭제하지 않고 저장
- 여러 consumer를 동시에 읽음
- 여러 번 읽을 수 있음

**장점**

- 대용량 처리
- 높은 확장성
- 로그/이벤트 처리 강함

**언제 사용?**

- 로그 수집
- 실시간 데이터 분석
- 이벤트 스트리밍

→ 데이터 흐름 자체를 처리하는 시스템

### Redis

인메모리 기반의 key-value 저장소

→ 메시지 큐는 아니지만, 관련 기능들을 제공하고 있어서 메시지 큐처럼 사용 가능

**특징**

- 데이터를 하드웨어(디스크)가 아니라 메모리(RAM)에 저장
- 속도가 아주아주 빠름
- key-value 데이터 구조에 기반한 다양한 형태의 자료 구조 제공
- PUB/SUB과 같은 기능으로 메시징도 가능하지만, 안정성이 좋지 않음
- 캐시 사용

**언제 사용?**

- 실시간 알림
- 채팅
- 간단한 이벤트 처리

→ 절대로 중요한 메시지를 처리할때는 사용하면 안됨

RabbitMQ vs. Kafka vs. Redis

- RabbitMQ
    
    → 작업 처리 (큐, 소비하면 삭제)
    
- Kafka
    
    → 데이터 저장 + 스트리밍 (읽어도 남음)
    
- Redis
    
    → 매우 빠른 처리 + but, 메시지 보장은 약함

## 💡 회고 및 질의응답
<aside>

**그렇다면 저 기술들이랑 대용량 트래픽 처리가 정확히 무슨 상관이냐?**

처음에는 단순히 비동기 처리라서 관련이 있는건가 싶었는데, 그렇게 보면 스프링의 @Async만 있어도 충분하지 않나라는 의문이 들었다.

하지만 중요한 차이는 **`어디에서 처리하는지!`** 

@Async는 서버 내부에서 스레드를 늘려 처리하는 방식이라, 결국에는 서버가 감당할 수 있는 한계를 넘으면 요청이 튕기게 된다.

메시지 큐는 요청을 외부 큐에 저장해두고 서버가 가져와서 처리하는 구조라, 요청이 몰려도 데이터가 유실되지 않고 순차적으로 처리된다. 부하를 외부로 빼는 구조라고 생각하면 된다.

즉, 단순히 비동기라서가 아니라, 트래픽이 몰릴 때 요청의 유실 없이 안정적으로 처리하기 위한 구조라는 점에서 대용량 트래픽 처리와 관련이 있다.

</aside>
