## 📝 주요 개념 정리
## 1. 통합 모니터링 시스템의 목적

통합 모니터링 시스템은 여러 애플리케이션과 인프라에서 발생하는 **로그(Log), 메트릭(Metrics), 트레이스(Trace)** 데이터를 수집하여 서비스 상태를 실시간으로 관찰하고 문제를 분석하기 위한 시스템이다.

현대 서비스 환경에서는 MSA, k8s 등 하나의 서비스가 여러 서버와 애플리케이션으로 구성되는 경우가 많다. 이러한 환경에서는 장애 발생 시 문제 원인을 빠르게 파악하기 어렵기 때문에, 시스템 전반의 상태를 통합적으로 관찰할 수 있는 모니터링 체계가 필요하다.

통합 모니터링 시스템을 구축하면 다음과 같은 기능을 수행할 수 있다.

- 서비스 상태 실시간 확인
- 성능 지표 분석
- 장애 발생 원인 추적
- 로그 기반 문제 디버깅
- 서비스 요청 흐름 추적

---

## 2. 모니터링의 주요 데이터 유형

통합 모니터링 시스템은 일반적으로 다음 세 가지 데이터를 중심으로 구성된다.

### 2.1 Logs

로그는 애플리케이션 내부에서 발생하는 이벤트 기록이다.

예시

- 사용자 로그인
- API 요청 처리
- 예외 발생
- 데이터베이스 쿼리 실행

로그는 문제 상황의 원인을 가장 자세하게 분석할 수 있는 데이터이다.

예시 JSON 로그

```json
{
  "@timestamp":"2026-03-15T12:00:00.123Z",
  "log.level":"INFO",
  "service.name":"auth-service",
  "message":"User login success",
  "trace.id":"018f1d41-9f61-7a3d-9f1b-9f9d4f1d3e1"
}
```

---

### 2.2 Metrics

메트릭은 시스템 상태를 수치 형태로 표현한 데이터이다.

대표적인 메트릭

- CPU 사용률
- 메모리 사용량
- 요청 처리 시간
- 초당 요청 수 (RPS)
- 에러율

메트릭은 시스템의 **전체적인 성능 상태를 빠르게 파악하는 데 사용된다.**

예시

```
cpu_usage = 72%
request_per_second = 320
error_rate = 0.2%
```

---

### 2.3 Trace

트레이스는 하나의 요청이 여러 서비스와 컴포넌트를 거쳐 처리되는 **전체 흐름을 추적하는 데이터**이다.

예시

```
Client
  → API Gateway
    → Auth Service
      → User Service
        → Database
```

트레이스를 사용하면 다음과 같은 분석이 가능하다.

- 어떤 서비스에서 지연이 발생하는지 확인
- 요청 처리 흐름 분석
- 병목 구간 파악
- latency waterfall

예시

```
TraceID: 8c3a1c...
AuthService        10ms
UserService        50ms
Database           20ms
Total latency      80ms
```

## 3. 통합 모니터링 시스템의 구성 요소

일반적인 통합 모니터링 시스템은 다음과 같은 구조로 구성된다.

```
Application
     │
     │ Logs / Metrics / Traces
     ▼
Telemetry Collector
     │
     ▼
Storage Backend
     │
     ▼
Visualization Dashboard
```

각 구성 요소의 역할은 다음과 같다.

---

### 3.1 Application

모니터링 데이터를 생성하는 애플리케이션이다.

예시

- Spring Boot 서비스
- Python 서비스
- Node.js 서비스

애플리케이션은 다음과 같은 데이터를 생성한다.

- 로그
- 메트릭
- 트레이스

---

### 3.2 Telemetry Collector

애플리케이션에서 발생한 데이터를 수집하는 역할을 수행한다.

대표적인 도구

- OpenTelemetry Collector
- Fluentd
- Logstash
- Prometheus

Collector는 다음 기능을 수행한다.

- 데이터 수집
- 데이터 변환
- 데이터 전달

---

### 3.3 Storage Backend

수집된 데이터를 저장하는 시스템이다.

데이터 유형별로 사용하는 스토리지가 다르다.

| 데이터 | 대표 스토리지 |
| --- | --- |
| Logs | Elasticsearch, Loki |
| Metrics | Prometheus |
| Traces | Tempo, Jaeger |

---

### 3.4 Visualization Dashboard

수집된 데이터를 시각화하는 도구이다.

대표적인 도구

- Grafana
- Kibana

이 도구를 통해 다음과 같은 기능을 제공한다.

- 로그 검색
- 메트릭 그래프 확인
- 트레이스 분석
- 알림(Alert) 설정

---

## 4. 통합 모니터링 시스템 동작 흐름

통합 모니터링 시스템의 기본 흐름은 다음과 같다.

```
1. 애플리케이션에서 로그 / 메트릭 / 트레이스 생성
2. Collector가 데이터를 수집
3. 수집된 데이터를 스토리지에 저장
4. Grafana 등의 대시보드에서 데이터를 시각화
5. 운영자가 시스템 상태를 확인
```

예시 흐름

```
Spring Boot Service
        │
        │ JSON Logs
        ▼
   Loki / Elasticsearch
        │
        ▼
      Grafana
```

---

## 5. 기대 효과

- 장애 대응 속도 향상
    - 문제 발생 시 로그, 메트릭, 트레이스를 통해 원인을 빠르게 분석할 수 있다.
- 시스템 성능 분석
    - 메트릭 데이터를 통해 서비스의 성능 상태를 지속적으로 확인할 수 있다.
- 서비스 안정성 향상
    - 이상 징후를 조기에 발견하고 대응할 수 있다.
- 운영 효율성 향상
    - 여러 서비스의 상태를 하나의 대시보드에서 확인할 수 있다.

## 6. Logging Schema

분산 시스템에서 하나의 대시보드 서비스로 로그를 수집하기 위해서는 기존의 멀티라인 형식의 로그가 아닌 공통된 형식의 로그 형식이 필요하며, 대표적으로 json이 있다.

<aside>

### ECS(Elastic Common Schema)

**정의**
```
ECS는 공통 필드 세트, Elasticsearch 데이터 유형, Elasticsearch로 수집된 데이터에 대해 허용되는 값 및 사용 계층 구조를 정의하는
커뮤니티 기반의 오픈 소스 사양입니다. ECS는 검색, 드릴다운, 피봇팅, 데이터 시각화, 머신 러닝 기반의 이상 징후 탐색, 탐지 규칙, 경보 등
Elastic에서 이용할 수 있는 모든 분석 모드를 통합합니다.
```

### Flat Json / Nested Json


**Flat Json**

```json
{
  "@timestamp": "2026-03-15T14:40:12Z",
  "log.level": "ERROR",
  "service.name": "likelion",
  "environment": "prod",
  "trace.id": "018f6c9b0b8c7d4cbb1fd1f8c5a0c321"
}
```

</aside>

<aside>

**Nested Json**

```json
{
  "@timestamp": "2026-03-15T14:40:12Z",
  "log": {
    "level": "ERROR"
  },
  "service": {
    "name": "likelion",
    "environment": "prod"
  },
  "trace": {
    "id": "018f6c9b0b8c7d4cbb1fd1f8c5a0c321"
  }
}
```

</aside>

- 두 가지 방식 모두 적용 가능
- json structure가 flatten할수록 조회 성능 우수
