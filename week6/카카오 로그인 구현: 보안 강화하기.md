## 📝 주요 개념 정리

### 1. 핵심 보안 용어 정리
- **XSS (Cross-Site Scripting):** 공격자가 웹 사이트에 악의적인 자바스크립트 코드를 삽입하여, 사용자의 브라우저에서 실행되게 함으로써 세션 토큰 등 민감한 정보를 탈취하는 공격 방식.
- **CSRF (Cross-Site Request Forgery):** 사용자가 자신의 의지와 무관하게 공격자가 의도한 행위(수정, 삭제, 등록 등)를 특정 웹사이트에 요청하게 만드는 공격 방식.

### 2. 인증 아키텍처의 진화 과정
프로젝트 초기에는 백엔드에서 로그인 처리를 담당해달라는 요청에 맞춰 GET 방식의 리다이렉트를 사용했으나(찾아본 결과, 개발 생산성은 더 좋다고...), SPA 구조로 고도화되면서 2번의 보완을 진행

**[1단계] GET 리다이렉트 방식의 한계**
초기 MVP 모델에서는 카카오 인증 후 백엔드가 토큰을 생성하여 프론트엔드로 강제 리다이렉트하는 방식을 사용함.
- **취약점:** JWT 토큰이 URL 쿼리 파라미터에 명시적으로 노출됨. 브라우저 방문 기록, 프록시 서버 로그 등에 영구적으로 기록되어 토큰 탈취의 직접적인 원인이 됨. -> 말도 안되는 방식으로, 표준 Post 방식으로 전환 결정

**[2단계] RESTful POST 통신 도입**
URL 노출 문제를 해결하기 위해 프론트엔드가 인가 코드를 받아 백엔드로 POST 요청을 보내는 API 기반 통신으로 전환함.
- **개선점:** 인가 코드와 발급된 토큰이 HTTP 응답 바디를 통해 전달되므로 URL 노출을 차단함. HTTPS 암호화 구간 내에서만 데이터가 전송되어 기밀성을 확보함.
- **한계점:** 발급받은 액세스 토큰과 리프레시 토큰을 모두 JSON 바디로 전달하면 프론트엔드는 이를 로컬 스토리지나 메모리에 저장해야 함. 이때 자바스크립트를 통한 XSS 공격에 수명이 긴 리프레시 토큰이 노출되는 취약점이 존재함.
  
**[3단계] 웹 표준 방어 기법 적용**
2단계의 한계를 보완하기 위해 OSI 7계층 중 애플리케이션 계층의 보안을 강화하는 두 가지 요소를 도입함.
- **1. 리프레시 토큰의 HttpOnly 쿠키 전환**
수명이 긴 리프레시 토큰은 자바스크립트가 접근할 수 없는 HttpOnly 속성의 쿠키로 발급함. - XSS 공격 방어
  - **도입 근거:** HttpOnly 쿠키는 OS 및 브라우저 엔진 단에서 자바스크립트의 접근을 차단하여 XSS 공격을 무력화함. 추가로 Secure 속성을 부여하여 TLS 암호화가 적용된 HTTPS 통신에서만 토큰이 전송되도록 강제함.
  - 추가로, 금융권 등 시스템 안정성을 최우선으로 하는 환경에서는 개발자의 커스텀 로직보다 브라우저의 자체 보안 표준을 활용하는 것이 필수적임.

- **2. State 파라미터 검증 도입**
해커가 피해자의 브라우저를 조종해 위조된 요청을 보내는 CSRF 공격은 정상 통신으로 위장하므로 하위 네트워크 계층에서 방어할 수 없음.
  - **도입 근거:** State 파라미터는 로그인을 시작한 브라우저와 최종 인가 코드를 전달하는 브라우저가 동일한지 검증하는 난수(무작위 추출값)임. 백엔드가 최초 발급한 난수와 콜백으로 돌아온 난수를 대조하여 요청의 무결성을 증명함.
  - 이는 전 세계 웹 보안 표준인 OAuth 2.0 프로토콜에 명시된 글로벌 표준 규격임.

### 3. AWS Cognito (고려사항)
AWS Cognito 같은 완전 관리형 서비스를 사용하면 복잡한 인증 로직과 토큰 관리를 인프라 단에서 자동으로 안전하게 처리 가능함.
- **도입X 근거:** AWS Cognito는 인증 인프라 구축을 생략하고 빠른 시장 출시가 필요하거나 단순한 표준 인증만 요구되는 서비스에 적합함.
- 하지만 현재 프로젝트는 카카오 로그인 시 유저 식별에 그치지 않고, 랜덤 프로필 이미지 할당 및 카테고리(클래시버스) 자동 잠금 해제 등 복잡한 도메인 DB 트랜잭션이 인증 과정에 강하게 결합되어 있음.
- Cognito를 도입할 경우 이러한 커스텀 로직 처리를 위해 AWS Lambda 트리거를 별도로 구성하여 애플리케이션 DB와 통신하게 만들어야 하므로 아키텍처가 불필요하게 복잡해짐.
- 또한 SPA 환경의 취약점을 막기 위해 수명이 짧은 액세스 토큰은 응답 바디로 전달하고, 리프레시 토큰은 HttpOnly 및 Secure 속성의 쿠키로 분리 전송하는 세밀한 헤더 제어가 필수적이나, 완전 관리형 서비스는 이러한 커스텀 응답 제어가 제한적임.
- 따라서 로그인 시 수반되는 복잡한 비즈니스 로직을 효율적으로 처리하고 웹 보안 표준을 직접 제어하기 위해서는 자체 인증 서버를 구축하는 것이 현재 서비스에 훨씬 적합하고 타당함.

## 💻 코드 실습 및 분석

### 1. 현재 구현 상태 (2단계 RESTful POST 통신 도입)
아래 `AuthController.java`의 로직은 프론트엔드로부터 인가 코드를 JSON 바디로 받아 처리하는 2단계임

```java
// AuthController.java 
// 프론트엔드 요청 대응용 POST
// 1. 요청 Body를 받을 DTO
public record KakaoLoginRequest(String authorizationCode) {}

// 2. 프론트엔드가 인가 코드를 직접 보내주면 토큰을 반환하는 POST 메서드
@PostMapping("/kakao/callback")
public ResponseEntity<UserAuthDto.AuthResponse> kakaoLoginPost(
        @RequestBody KakaoLoginRequest request
) {
    // 기존 서비스 로직 재사용
    UserAuthDto.AuthResponse response = authService.loginWithKakao(request.authorizationCode());
    return ResponseEntity.ok(response);
}
```
- 문제점 분석: authService.loginWithKakao 로직을 거쳐 반환된 AuthResponse에는 액세스 토큰과 리프레시 토큰이 모두 포함되어 있음. 리프레시 토큰이 응답 바디로 그대로 노출되어 프론트엔드 환경에서 XSS 공격에 탈취될 위험이 큼. 또한 State 난수 검증 로직이 누락되어 CSRF 공격에 취약함.

### 2. 3단계 보안 표준 적용
```java
// 1. State 파라미터가 추가된 Request DTO
public record KakaoLoginSecureRequest(String authorizationCode, String state) {}

@PostMapping("/kakao/callback/secure")
public ResponseEntity<AccessTokenResponse> kakaoLoginSecurePost(
        @RequestBody KakaoLoginSecureRequest request,
        HttpServletResponse response
) {
    // 2. State 값 무결성 검증 (사전에 세션이나 Redis 등에 저장해둔 state와 비교)
    authService.validateState(request.state());
    
    // 3. 서비스 로직에서 토큰 발급
    TokenDto tokens = authService.loginWithKakao(request.authorizationCode());

    // 4. Refresh Token을 Response Body가 아닌 HttpOnly 쿠키로 주입
    ResponseCookie refreshTokenCookie = ResponseCookie.from("refresh_token", tokens.getRefreshToken())
            .httpOnly(true)
            .secure(true) // HTTPS 통신 강제
            .sameSite("Strict") // CSRF 추가 방어
            .path("/")
            .maxAge(14 * 24 * 60 * 60)
            .build();
    
    response.addHeader(HttpHeaders.SET_COOKIE, refreshTokenCookie.toString());

    // 5. 탈취되어도 피해가 적은 수명이 짧은 Access Token만 Body로 반환
    return ResponseEntity.ok(new AccessTokenResponse(tokens.getAccessToken(), tokens.getNickname()));
}
```

## 💡 회고 및 질의응답
- 기능 구현에 집중하여 보안의 거시적인 관점을 챙기지 못했다. API가 정상적으로 작동 여부를 넘어서, 실제 브라우저 환경에서 악의적인 자바스크립트 코드가 동작할 가능성과 요청 위조 가능성을 차단하는 설계의 중요성을 깨달았다.

**Q. 리프레시 토큰만 쿠키에 담고 액세스 토큰은 여전히 바디로 넘겨주는 이유?**
- 액세스 토큰까지 쿠키에 담을 경우 모든 API 요청 시 브라우저가 자동으로 해당 쿠키를 전송하게 됨. 이는 XSS 공격은 방어하지만 CSRF 공격에 완전히 노출되는 결과를 초래함. 수명이 짧은 액세스 토큰은 메모리에 저장하여 명시적으로 헤더에 담아 보내고, 수명이 긴 리프레시 토큰만 HttpOnly 쿠키로 격리하는 것이 현대 웹 보안의 권장 패턴임.

**Q. State 검증을 생략할 경우 발생하는 구체적인 보안 위협은 무엇인가?**
- 공격자가 본인 계정의 인가 코드를 얻어낸 뒤 해당 코드가 포함된 URL을 피해자에게 전송함. 피해자가 링크를 클릭하여 POST 요청이 발생하면 백엔드는 이를 피해자의 로그인 세션으로 인식하고 공격자 계정과 연동함(로그인 CSRF 공격). State 난수 검증이 없으면 백엔드 애플리케이션 계층에서 정상 요청과 위조된 요청을 판별할 수 없음.
