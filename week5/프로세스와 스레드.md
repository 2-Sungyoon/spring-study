## 📝 주요 개념 정리

### 1. 백엔드 환경에서의 프로세스와 스레드
- 프로세스(JVM): 자바로 작성된 스프링 애플리케이션을 실행하면 운영체제 위에서 하나의 JVM 프로세스로 동작한다. 이 프로세스는 힙과 같은 독립적인 메모리 공간을 할당받아 애플리케이션의 데이터를 관리한다.
- 스레드(Tomcat Worker Thread): 스프링 부트 내장 톰캣은 HTTP 요청이 들어올 때마다 프로세스 내부의 스레드를 하나씩 할당하여 요청을 처리한다. 여러 사용자가 동시에 접속하면 여러 스레드가 동시에 생성되어 병행 처리를 수행한다.

### 2. 스레드 풀과 컨텍스트 스위칭 오버헤드
- 스레드를 무한정 생성하면 스레드 간 전환(Context Switching) 비용이 기하급수적으로 증가하여 CPU가 실제 로직 처리보다 스레드 교체에 더 많은 자원을 낭비하게 된다.
- 이를 방지하기 위해 톰캣은 미리 정해진 개수의 스레드만 만들어두고 재사용하는 스레드 풀 방식을 사용한다. 데이터베이스 커넥션을 미리 맺어두는 커넥션 풀도 동일한 병목 해결 원리다.
- 병렬성 (Parallelism):
    - 정의: 여러 개의 물리적 코어(Core)나 프로세서를 사용하여 작업을 동시에 처리.
    - 방식: 작업을 여러 개로 나누어 각 코어에 분배하여 1초에 동시에 처리.
    - 핵심: 물리적인 동시성 (진짜 동시에 실행)
- 병행성 (Concurrency):
    - 정의: 하나의 코어에서 여러 작업이 교차하며 처리되는 방식.
    - 방식: CPU가 아주 짧은 시간 동안 작업을 번갈아 가며 처리(Context Switching)하여, 사용자는 동시에 처리되는 것처럼 느끼게 함.
    - 핵심: 논리적 동시성 (동시 실행처럼 보임)

### 3. 스프링 싱글톤 빈과 동기화 문제
- 톰캣의 여러 스레드는 JVM의 힙 메모리를 공유한다.
- 스프링의 컴포넌트(컨트롤러, 서비스 등)는 기본적으로 단일 객체인 싱글톤으로 생성되어 모든 스레드가 이를 공유한다.
- 싱글톤 빈 내부에 변경 가능한 필드 변수를 두면, 여러 스레드가 동시에 접근할 때 값들이 충돌하는 레이스 컨디션이 발생한다.

### 4. IPC와 마이크로서비스(MSA)
- 멀티 스레드는 메모리를 공유하므로 빠르지만, 하나의 스레드에서 발생한 치명적 오류가 프로세스 전체를 다운시킬 수 있다.
- 이를 막고 확장성을 높이기 위해 백엔드 아키텍처는 단일 거대 프로세스에서 여러 개의 작은 프로세스로 나누는 MSA로 발전하고 있다.
- 나뉘어진 프로세스들은 메모리를 공유할 수 없으므로, 네트워크 소켓을 이용한 HTTP REST API나 메시지 큐(Kafka, RabbitMQ) 같은 IPC 방식을 통해 데이터를 주고받는다.

## 💻 코드 실습 및 분석
1. 문제 상황: 상태를 가진 싱글톤 빈 (Stateful)
스프링 빈은 기본적으로 싱글톤으로 관리된다. 따라서 멤버 변수는 모든 스레드가 공유하는 Heap 영역에 저장되므로, 동시 수정 시 데이터가 덮어씌워지는 문제가 발생할 수 있다.

```java
@Service // 스프링 컨테이너가 이 객체를 하나만 생성해서 공유함 (싱글톤 방식)
public class StatefulOrderService {

    // 문제: 멤버 변수는 Heap 영역에 저장되어 모든 스레드가 공유함.
    // 여러 스레드가 동시에 값을 읽고 쓸 수 있어 레이스 컨디션 발생.
    private int orderCount = 0;

    public void createOrder() {
        // 1. 현재 공유 변수 값을 조회 (예: 스레드 A, B 모두 0을 읽음)
        int currentCount = orderCount;
        
        try {
            // CPU 제어권을 넘기기 위해 강제 대기 (동시성 문제)
            Thread.sleep(100); 
        } catch (InterruptedException e) {}
        
        // 2. 읽어온 값에 1을 더해 저장.
        // 스레드 A가 1을 저장하고, 스레드 B도 1을 저장해버림 -> 주문은 2건인데 카운트는 1만 증가 (데이터 유실)
        orderCount = currentCount + 1; 
    }
    
    public int getOrderCount() {
        return orderCount;
    }
}
```
2. 해결 방안: 무상태 설계 (Stateless)
동시성 이슈를 피하기 위해 스프링 빈은 무상태로 설계해야 한다. 공유되는 필드 변수 대신, 지역 변수(Stack 영역)를 사용하거나 상태 관리를 DB에 위임한다.
```java
@Service
public class StatelessOrderService {
    
    // 상태(데이터)를 가지지 않고, DB 처리를 위한 불변 객체(Repository)만 주입받음
    private final OrderRepository orderRepository;
    // 생성자를 통한 해결
    public StatelessOrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    public void createOrder() {
        // 지역 변수는 각 스레드의 Stack 영역에 별도로 생성됨
        // 스레드끼리 침범할 수 없으므로 동시성 문제를 해결
        Order order = new Order();
        
        // 상태 관리는 애플리케이션 메모리가 아닌, 트랜잭션이 보장되는 DB에 위임
        orderRepository.save(order);
    }
}
```

## 💡 회고 및 질의응답
- Q: Node.js는 싱글 스레드인데 어떻게 수많은 동시 요청을 처리하는가?

A: Node.js는 단일 스레드로 동작하지만, 무거운 I/O 작업을 백그라운드의 워커 스레드나 운영체제 커널에 비동기로 위임하고 즉시 다음 요청을 받는다. 이를 이벤트 루프 기반의 논블로킹 I/O라고 하며, 컨텍스트 스위칭 오버헤드가 적어 I/O가 많은 웹 서버에 효율적이다. 자바 진영에서도 이를 차용해 Spring WebFlux를 도입했다.

- Q: 최근 자바 21에 도입된 가상 스레드(Virtual Thread)는 기존 스레드와 무엇이 다른가?

A: 기존 자바의 스레드는 운영체제의 스레드와 1:1로 매핑되어 생성 개수에 제한이 있고 스위칭 비용이 무거웠다. 가상 스레드는 JVM 내부에서 관리하는 경량 스레드로, 하나의 운영체제 스레드에 수많은 가상 스레드를 다대일로 매핑하여 동작한다. 기존의 동기식 코드를 그대로 유지하면서도 비동기 수준의 처리량을 확보할 수 있다.
